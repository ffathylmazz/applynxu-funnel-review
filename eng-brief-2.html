Accept the ID people actually have — client-side file normalization + Front/Back

Owner Fatih (PM)Size ~M · frontend (lazy-loaded libs)Priority P1 — step nearest to payment

Problem — why this matters

ID and selfie uploads accept only .png/.jpg/.jpeg ≤10MB. But this audience's ID is a PDF scan, an iPhone HEIC, or a 12MB phone photo — each rejected via a small corner toast, at the step nearest to payment. And a one-sided ID is accepted (front/back isn't enforced), which can bounce back after payment as a re-upload → refund/support/churn.

Reproduced live: id.pdf → "The following files are not supported" · 11MB JPEG → size rejection · single image → accepted through to checkout (201, async verification)

Why build this — expected effect: ≈ +4–10% App start → App complete. The biggest confirmed late-funnel blocker, at the step nearest payment — and it stops post-payment re-upload churn (a won conversion turning into a refund). Sized on an assumed baseline in the impact model (full report, §12); the numbers that replace the assumption: Step-3 rejection reasons + first-try acceptance rate.

Acceptance criteria

Scenario PDF auto-converted
Given I select a PDF When it’s added Then page 1 (and page 2 → front/back) becomes an image with a preview; no error

Scenario Oversized / HEIC handled
Given a >10MB or .heic photo When added Then it’s down-scaled/re-encoded to a compliant JPEG and accepted

Scenario Front & back enforced
Given a two-sided ID type When I continue with one image Then I’m blocked with an inline message (passport = one slot)

Scenario Unconvertible file fails usefully
Given a corrupt/locked file or one that can’t reach 10MB legibly When processing fails Then the slot shows a specific, actionable error — never a generic rejection

Scope

In scope — build this

  • Client-side convert/compress pipeline (PDF → image, HEIC decode, resize/re-encode)
  • Front/Back slots with document-type awareness, enforced pre-payment
  • Inline slot states replacing the corner toast

Not in scope — don’t touch

  • ID content — normalization only (format/size); no cropping, enhancement or alteration: KYC/anti-fraud integrity untouched
  • The server’s 10MB / image-only contract (it feeds the face-match/OCR pipeline — that rule is correct)
  • The transcript step (accepts PDF today; see open questions)

Dependencies · rollout · success

Depends onpdf.js + canvas re-encode + a HEIC decoder — lazy-loaded on the identity step only (don’t add weight to register; see the Perf finding). KYC pipeline expects images (confirmed). Server already combines front+back.
RolloutFeature flag per file-type (PDF → HEIC → oversize) so any KYC-pipeline regression is isolable and reversible. Watch verification pass-rate in week 1 (A/A check on converted vs. native images).
SuccessFirst-try upload acceptance ↑ · Step-3 format/size rejections ↓ materially · no OCR/face-match quality regression on converted images · fewer post-payment re-upload requests.
