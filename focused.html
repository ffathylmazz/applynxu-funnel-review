Three changes that move the number.

A focused read: the three highest-leverage changes to grow App start → App complete, each grounded in the real walkthrough. The full ~30-finding analysis and a clickable prototype are linked at the end.

The only metricApp start → App complete (pay the $25 fee)

Why these three — and what I left out

I walked the whole application on desktop and mobile and logged ~30 findings. Rather than list them all, this page is the three highest-leverage changes, deliberately spread across the effort/risk range: one cheap confirmed fix (C1), one medium confirmed fix (file handling), and one big bet (KYC timing). The pattern across all ~30: failure severity increases as the applicant invests more — cosmetic at register, submission-blocking by identity — the opposite of a healthy funnel.

Everything else — local payment rails, validation UX, performance, the off-funnel recovery loop, the full measurement plan — is real but secondary, and lives in the full analysis (linked below). Two items I found (broken conversion tracking, an OAuth token in the URL) aren't conversion levers; they're labelled as such there.

What's already working — keep these

Passwordless, one-screen sign-up · honest "<10 min" promise · one-tap Google
Document lists localize by country (NIMC ID, Voter's ID) · WhatsApp verification
Saved & resumable — returns you to your last step, nothing re-entered
A human recovery loop (email always + call + WhatsApp) already wins back drop-offs
Sponsor invoicing · country-adjusted pricing · easy program-switch, no restart

Prerequisite — fix the tracking first (C2). Today conversion pixels fire on failed sign-ups, so “App starts” are partly fictional. Gate every pixel on a 2xx success before trusting any number below — you can't optimize a metric you're measuring wrong.

01

End the dead-ends

● CriticalConfirmed

Before — today

After — proposed

Phone number
🇺🇦 +380 · 95 849 9398
⚠ Nexford can't accept phone numbers from Ukraine yet. Try another number, or see supported countries.

What happens

The phone field offers 245 countries, but the backend rejects some codes (e.g. Ukraine +380). A fully completed form returns only “An error occurred. Please try again.” — no field, no reason, no way forward. Evidence: 400 on /validate — “PhoneCountryCode 380 … condition not met”.

Why it costs App-complete

  • A hard loss at the widest, highest-intent point — the applicant already filled everything.
  • Silent: it never shows in a support queue — the applicant just leaves. It also contradicts the “100+ countries” promise.

The change

  • Don't offer unsupported countries (or flag them at selection); return a field-level, actionable message.
  • The country limit itself may be intentional — the silent generic error is not.
≈ +2–6% completes — high-intent recovery (they finished the form).
How we'd know: register drop by country (400s ÷ attempts). Cheapest fix, widest point → ship first.

Same class, deeper in the funnel — re-verify

In a separate test run (12 Jul), answering “No” to “Are you a native English speaker?” dead-ended the form the same way: the required English-Proficiency field never renders, so submit fails with an error no one can satisfy. For a 100+-country school the “No” segment is plausibly 25–40% of applicants — potentially the single largest leak. Same fix principle: validation may only require rendered fields. Re-verify, then size via % of “No” answers.

02

Accept the ID people actually have

● HighConfirmed

What happens

ID and selfie uploads accept only png/jpg ≤10MB. A PDF scan is rejected via a small corner toast; an 11MB photo fails for size; iPhone HEIC would fail too. A one-sided ID is accepted — front/back isn't enforced. This is the step nearest to payment.

Why it costs App-complete

  • This audience's ID is often a PDF scan, a HEIC, or a large photo — and they can't convert files easily.
  • An incomplete ID may bounce back after payment as a re-upload — turning a won conversion into a refund/churn.

The change

  • Normalize on the client — PDF→image, HEIC decode, auto-compress — so the KYC pipeline still gets a clean image and the user does nothing.
  • Enforce Front + Back (document-type-aware) before payment. Full spec in the brief below.
≈ +4–10% completes — removes the biggest confirmed late-funnel blocker.
How we'd know: Step-3 rejection reasons + first-try acceptance rate.
03

Move KYC to after payment

● Biggest betValidate

What happens

Identity documents + selfie — the heaviest step — sit before the $25 payment. The applicant must clear a full KYC gate before committing a cent.

Why it costs App-complete

  • Front-loading heavy verification before payment likely drops the most people at the highest-intent moment.
  • But it may be deliberate fraud/quality gating — so this is a bet, not a mandate.

The change

  • Move identity verification to after payment (or to admission) as a bounded A/B with a kill switch.
  • Sign-off: Compliance/Risk + Finance + exec; ship only if App-complete rises with fraud/refund flat.
  • Benchmark during validation: how comparable online universities (UoPeople, WGU, SNHU, Quantic) sequence identity checks — commonly at/after enrolment rather than before the application fee.
≈ +8–20% completes (fraud-gated) — potentially the single biggest lift.
How we'd know: A/B move-after-payment; watch fraud, refunds, no-shows.

Engineering brief · Change 01

End the “unsupported country” dead-end

Also available as a standalone engineer hand-off doc → with 1-click links into the prototype.

As an applicant from any country Nexford serves, I want to know upfront if my phone country is supported and get a clear message if not — so I can finish instead of hitting an unexplained error.

Scenario Unsupported country is prevented at the source
Given my country isn't on the supported list
When I open the phone-country selector
Then it's shown disabled with a short reason — I can't reach a state that will 400

Scenario Any validation error is field-level
Given the backend rejects a field (e.g. PhoneCountryCode)
When the response returns
Then the error shows inline on that field, the page focuses it, and the generic toast is gone

Scenario Pixels never fire on failure
Given a non-2xx submit When it's handled Then no lead/conversion pixels fire

Scenario Validation only requires rendered fields
Given a conditional rule is active When its field is not rendered Then the rule cannot block submit (regression test covers the English-proficiency case)
Edge casesempty (client-validate first), loading (disable + spinner, no double-submit), 5xx (retryable banner), mobile (error scrolls into view), Google path enforces the same rule
Depends oncanonical supported-country list (Admissions/Compliance); validate endpoint already returns a field key; move pixel calls into the success callback
Successregister-completion for affected countries rises toward the accepted-country baseline; zero generic error toasts; pixels verified to fire only on 2xx
Rolloutbehind a feature flag — field-level error first, then the disabled-country list; watch register drop by country for 2 weeks
Non-goalschanging which countries are supported — that's a policy decision (Admissions/Compliance), not this ticket
Open questionsIs the limit sanctions/ops or a bug? Hide vs. disabled-with-reason? Is phone required at register at all?
