End the register dead-end — unsupported phone country

Owner Fatih (PM)Size ~S–M · frontend + tag changePriority P1 — widest funnel point

Problem — why this matters

The phone selector offers 245 countries but the backend rejects some. The user fills the whole form, hits Register, and gets only a generic toast: “An error occurred. Please try again.” No field, no reason, no way forward — a silent loss at the funnel's widest point. Conversion pixels still fire on the failed submit.

Evidence: POST /learner-profile/…/validate → 400 · "PhoneCountryCode 380 … condition not met" · reproduced with 🇺🇦 +380

Why build this — expected effect: ≈ +2–6% App start → App complete. Recovers high-intent users who already completed the form — the cheapest fix at the widest funnel point. Sized on an assumed baseline in the impact model (full report, §12); the number that replaces the assumption: register drop by country (400s ÷ attempts). Surface 3 (2xx-gated pixels) is what makes the effect measurable at all.

Acceptance criteria

Scenario Unsupported country is prevented at the source
Given my country isn’t on the supported list
When I open the phone-country selector
Then it’s shown disabled with a short reason — I can’t reach a state that will 400

Scenario Any validation error is field-level
Given the backend rejects a field (e.g. PhoneCountryCode)
When the response returns
Then the error shows inline on that field, the page focuses it, and the generic toast is gone

Scenario Pixels never fire on failure
Given a non-2xx submit When it’s handled Then no lead/conversion pixels fire

Scenario Validation only requires rendered fields
Given a conditional rule is active When its field is not rendered Then the rule cannot block submit

Scope

In scope — build this

  • Selector filtering from the canonical supported-country list
  • Backend field-key → inline error mapping + focus/scroll
  • Move all lead/conversion tags into the 2xx callback

Not in scope — don’t touch

  • Which countries are supported — policy call (Admissions/Compliance), not this ticket
  • Register-page layout / visual redesign
  • Phone-format validation rules beyond country support
